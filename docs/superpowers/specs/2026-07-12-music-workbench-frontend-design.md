# 音乐工作台前端改造设计

## 目标

把现有 GUI 从通用的卡片式管理面板改造成具有音乐产品辨识度的桌面工作台，同时保留全部搜索、选择、下载、重试、环境检查和日志能力。

目标用户是在 Windows、macOS 或 Linux 桌面上使用本工具搜索并批量下载音乐的人。页面的首要任务是让用户尽快完成“输入关键词 → 查看结果 → 选择歌曲 → 下载”。

## 已确认的范围

- 允许明显重排界面，但不改变 Python 桥接接口、参数或业务语义。
- 默认窗口尺寸改为 `1280x800`，最低窗口尺寸改为 `1024x720`。
- `1280x800` 及以上使用结果区加右侧活动栏布局；较窄窗口把活动栏移到结果区下方。
- 搜索成为首要操作，常用设置靠近搜索，其余设置可展开。
- 日志默认折叠，下载状态始终清晰可见。
- 修复已识别的焦点、文本选择、动态播报、减少动效、图片尺寸和表单细节问题。
- 更新所有相关文档、测试、代码注释和构建后的 GUI 静态产物，不能只更新 README。

## 非目标

- 不修改 `window.pywebview.api` 的方法集合和数据结构。
- 不修改搜索、下载、重试、文件命名或元数据写入语义。
- 不引入外部字体、远程图标或新的前端依赖。
- 不增加深色主题、播放功能、歌曲预览或新的音乐源。

## 视觉方向

采用浅色“音乐工作台”，避免每个区域都是独立白色圆角卡片。页面使用更少、更明确的结构层级：顶部指挥区、结果工作区和活动区。

### 设计令牌

| 名称 | 色值 | 用途 |
|---|---|---|
| 画布 | `#F3F7FC` | 应用背景 |
| 主文字 | `#102033` | 标题与正文主色 |
| 音轨蓝 | `#2563EB` | 主操作、焦点和进度 |
| 波形青 | `#38BDF8` | 音轨视觉点缀 |
| 成功绿 | `#10B981` | 成功和环境通过状态 |
| 分隔线 | `#D9E4F0` | 面板和列表边界 |

字体角色：

- 标题和主要操作使用 `Microsoft YaHei UI` 及系统中文无衬线字体，并提高字重。
- 正文使用系统无衬线字体，保证跨平台可读性。
- 时间、进度、音质和计数使用 `Bahnschrift`、`Cascadia Mono`、`Consolas` 回退链，并启用等宽数字。

### 标志性元素

顶部使用一条“音轨线”连接品牌标识、搜索区与状态提示。下载进行时，进度条沿用同一套音轨蓝与波形青语言。除这一个标志性元素外，不增加无业务意义的装饰。

### 视觉细节补充

- 所有下拉框继续使用原生 `<select>`，以保留系统菜单、键盘操作和辅助技术语义；控件右侧统一使用不可交互的 `ChevronDown` 图标。支持 `select:open` 的环境在菜单展开时把该图标旋转为向上，未支持时保持向下而不影响选择行为。
- 主界面使用固定外壳并禁用窗口级滚动。宽屏四边保持 `16px`、低于 `1180px` 时四边保持 `12px` 等距边界；“更多设置”展开后由下方工作区让出高度，结果列表、日志和窄屏工作区在内部滚动，避免滚动条改变外层对齐关系。
- 启动页是短暂的品牌过渡，而非海报：缩小图标、标题、状态文字和进度条，降低并下移底部波纹，让状态信息保持清晰的第一阅读顺序。
- 宽屏活动栏以半透明浅色容器把下载进度和运行日志归为同一个活动区；窄屏移动到结果区下方时移除该外壳，保持纵向阅读轻盈。
- 搜索结果采用紧凑音乐库表格：`60px` 行高、`40px` 封面，歌曲/歌手、专辑和来源/状态按内容加权共享弹性空间，时长使用 `64px` 固定轨道并右对齐。选中行使用浅蓝底与左侧 `3px` 品牌色内描边，键盘焦点保留整行轮廓；结果总数和已选数合并在面板标题区，下载按钮保持静态“下载选中”文案以避免操作区横向位移。
- 结果行外围焦点框只在复选框触发 `:focus-visible` 时显示，鼠标取消选择后不保留整行轮廓。空状态不设置超过内容区的固定最小高度，始终相对结果卡片当前可用高度居中。
- Windows 窗口、任务栏和打包 EXE 使用与工作台品牌标记一致的蓝底白色音符多尺寸图标；源码与打包运行共享 `music_downloader/gui/assets/music_downloader.ico`。
- 窗口关闭操作始终显示与工作台一致的 Svelte 主题弹窗，不使用 Windows 原生 `MessageBox`。原生锁定式 `closing` 回调首次返回 `False` 以取消关闭，浏览器事件从后台线程发送，避免 GUI 线程与 `evaluate_js` 相互等待；确认接口先返回桥接结果，再异步销毁窗口，确认后的回调返回 `True`，避免访问已释放的 WebView2。弹窗只显示“关闭音乐下载器？”、“确定要关闭应用吗？”以及“继续使用”/“关闭应用”，初始焦点落在无轮廓的说明文本；`Esc`、遮罩和安全按钮均取消，确认后才获得一次性原生关闭许可并执行清理。搜索输入框使用 `aria-label="搜索关键词"` 暴露稳定的可访问名称。

## 信息架构与布局

宽窗口布局：

```text
┌ 品牌 + 音轨状态 ───── 搜索框 ───── 搜索按钮 ┐
│ 音源  类型  结果数量               [更多设置] │
├──────────────────────────┬───────────────┤
│ 搜索结果 / 批量操作       │ 下载状态       │
│                          │ 当前任务       │
│ 歌曲行                   ├───────────────┤
│ 歌曲行                   │ 运行日志       │
│ 歌曲行                   │ 默认折叠       │
└──────────────────────────┴───────────────┘
```

响应式规则：

- `1180px` 及以上：结果区与约 `320px` 活动栏并排。
- 低于 `1180px`：活动栏移到结果区下方，结果列表优先获得可用高度。
- 最低支持 `1024x720`；所有区域通过 CSS Grid/Flex 和内部滚动适配，不使用 JavaScript 测量布局。
- 批量操作可以换行，但不能覆盖歌曲标题、时长或选择框。

## 组件职责

### `App.svelte`

继续负责 pywebview 初始化、配置和歌曲状态、搜索、下载任务生命周期以及顶层响应式布局。现有桥接调用和事件处理保持不变。

### `SearchBar.svelte`

保留语义化搜索表单和回车提交，成为顶部最显眼的操作。搜索中状态使用明确文案并通过动态状态区域播报。

### `SettingsPanel.svelte`

改为两层设置：音源、类型和结果数量作为常用快捷设置；音质、封面、歌词、下载目录和环境检查放入可展开的更多设置区域。音源、类型、结果数量、音质和下载目录使用统一 `.field-stack`，标签与控件之间保持 `10px` 净空。设置仍在每次变更时通过现有回调保存。

### `ResultList.svelte`

保留列表式批量工作流，优化标题、歌手、专辑、来源、音质、下载状态和时长的视觉层级。批量操作位于列表标题栏，封面保留固定尺寸占位。

### `DownloadProgress.svelte` 与 `LogPanel.svelte`

共同组成活动区。下载进度使用音轨视觉语言并支持动态播报；日志默认折叠，展开后内容可选择和复制。错误仍进入运行日志，不改变后端成功语义。

### `StartupScreen.svelte`

保留已批准的品牌启动页和文案，只增加减少动效支持，并让色彩令牌与主界面一致。

## 交互与状态

- 初始化、搜索和下载期间继续禁用会产生冲突的操作。
- 搜索框始终可被键盘访问，并支持回车提交。
- 更多设置的展开状态只属于当前运行，不持久化到用户目录。
- 日志默认折叠；用户可手动展开或折叠，不因普通信息自动抢占焦点。
- 下载开始、单曲完成、取消和任务完成继续使用现有进度事件更新。
- 空状态继续引导用户输入关键词，不使用纯装饰性文案。
- 错误信息继续说明发生的问题；能够给出下一步时，应同时提供重试或检查入口。

## 可访问性基线

- 输入框、选择框和文本域通过 `:focus` 显示品牌蓝边框与柔和光晕；按钮、折叠项、复选框和单选框通过 `:focus-visible` 显示清晰的 `2px` 键盘焦点环。
- 表单控件保留可点击标签，并增加稳定的 `name`；非认证字段使用 `autocomplete="off"`。
- 搜索状态和下载进度使用 `aria-live="polite"`，避免对日志逐条强制播报。
- 装饰图标使用 `aria-hidden="true"`；图标按钮必须有可访问名称。
- 仅纯装饰元素和按钮禁用文本选择；歌曲信息、下载目录、错误原因和日志可选择复制。
- 所有动画和过渡在 `prefers-reduced-motion: reduce` 下关闭或降级。
- 封面图片声明 `width`、`height`，使用 `loading="lazy"` 和空 `alt`，因为歌曲文字已经提供等价信息。
- 文案使用中文省略号 `…`，计数和进度使用等宽数字。

## 窗口与桌面集成

- `music_downloader/gui/app.py` 的默认窗口改为 `(1280, 800)`。
- `music_downloader/gui/settings.py` 的 `DEFAULT_CONFIG` 必须使用同一默认尺寸，并由跨模块测试防止两处再次失配。
- 最低窗口改为 `(1024, 720)`。
- 仍使用 pywebview 本地窗口，不新增本地 HTTP 后端。
- 文件夹选择、打开目录、关闭时清理和浏览器会话行为保持不变。

## 测试设计

测试遵循先失败、后实现的顺序：

1. 增加前端工作台结构测试，要求搜索优先、更多设置、响应式活动区、可复制区域和动态状态语义存在。
2. 扩展启动页测试，要求提供 `prefers-reduced-motion` 降级。
3. 更新 Python GUI 应用测试，要求默认尺寸为 `1280x800`、最低尺寸为 `1024x720`。
4. 运行 `node --test tests/*.test.mjs`、`npm run check` 和 `npm run build`。
5. 运行相关 pytest，至少覆盖 GUI 应用、静态产物和构建脚本约定。
6. 使用模拟 pywebview 桥接分别在 `1024x720` 和 `1280x800` 渲染，检查溢出、焦点、设置展开、搜索结果和活动区位置。

## 验收标准

- 所有现有 GUI 功能和 pywebview API 保持可用。
- 搜索是进入主界面后的首要操作，常用与高级设置层级清晰。
- `1024x720` 无水平溢出，结果、下载状态和日志均可访问。
- `1280x800` 及以上展示并排活动栏。
- 键盘焦点清晰，日志和路径可复制，动态状态可被辅助技术感知。
- 减少动效偏好得到尊重，封面图片具备稳定尺寸和懒加载。
- Node 测试、Svelte 检查、前端构建和相关 pytest 全部通过。
- 所有相关文档、注释、测试期望和静态构建产物与新设计一致。
