# 音源目录统一与可靠性优化设计

日期：2026-07-13

## 背景与目标

本次改动处理项目质量复查中确认的六项问题，同时避免为了“优化”而引入不必要的抽象或兼容性风险：

1. GUI 下载后台任务异常时可能不发送完成事件，也不会从任务表移除。
2. 浏览器初始化失败时可能遗留 Playwright persistent context，锁住 `.chrome-profile` 并导致重试继续失败。
3. 下载接口返回 `200` 的 HTML、JSON 或 XML 错误页时，现有逻辑可能把它保存成 MP3/FLAC。
4. 音源 ID、可用列表和展示名散落在 Python 与前端多处，且当前遗漏酷狗音乐。
5. GUI 搜索关键词为空或搜索失败时，主要反馈只进入默认折叠的日志，用户不容易发现。
6. 搜索完成和结果数量变化缺少明确的辅助技术播报。

目标是修复上述行为、建立一份音源目录作为全局事实来源，并保持既有 CLI 参数、上游 API 值和下载成功语义不变。

## 非目标

- 不改变上游 API 的音源 ID，也不把中文展示名传给 API。
- 不为每个音源重新实现或探测上游搜索能力。
- 不使用严格的音频解码或 Mutagen 容器校验作为下载成功前置条件。
- 不改变“音频文件落盘成功即成功；标签、歌词和封面失败仅 warning”的语义。
- 不把音源选择持久化到用户目录。
- 不重构与六项问题无关的 GUI 布局、下载流程或业务分层。

## 音源审计结论

对当前上游站点脚本的音源定义和分支逻辑进行核对后，项目应支持 13 个可搜索音源。现有 12 个音源之外遗漏了 `kugou`；上游脚本存在该音源节点和专用处理逻辑，因此应加入全局目录。

| API 值 | 展示名 |
| --- | --- |
| `netease` | 网易云音乐 |
| `migu` | 咪咕音乐 |
| `kugou` | 酷狗音乐 |
| `kuwo` | 酷我音乐 |
| `ytmusic` | YouTube Music |
| `tidal` | Tidal |
| `qobuz` | Qobuz |
| `deezer` | Deezer |
| `spotify` | Spotify |
| `tencent` | QQ音乐 |

审计中还发现上游的专辑音源白名单没有列出 `kugou`。因此本项目将酷狗作为普通搜索音源加入，但不承诺上游一定支持酷狗专辑搜索。`embeat` 是上游内部推荐用途，`bilibili` 仅出现在代理等特殊逻辑中，二者不作为用户可选搜索音源。

## 方案选择

采用保守的“Python 枚举单一事实来源”方案。

`music_downloader/domain/enums.py` 中的 `Source` 同时保存稳定 API 值和展示名，例如成员逻辑上等价于 `NETEASE = ("netease", "网易云音乐")`。枚举提供只读 `label` 属性，并提供一个接收原始字符串的展示名解析函数；未知值回退为原始值，避免上游临时返回新音源时界面空白或报错。

其他位置只派生或消费这份目录：

```text
Source 枚举（值 + 展示名）
  ├─ core/config.py：派生 VALID_SOURCES
  ├─ CLI：参数校验仍使用原始值，帮助、提示、表格和日志使用展示名
  ├─ GUI API：返回 [{ value, label }] 音源选项
  ├─ GUI 搜索/下载日志：使用展示名
  └─ Svelte：使用 GUI API 返回的选项映射结果行，不维护本地音源表
```

这样既只有一份人工维护的目录，又不把 Python 业务定义复制到 TypeScript。前端构建产物仍由 Vite 生成，不直接修改 `gui/static/`。

## 详细设计

### 1. GUI 下载任务生命周期

下载后台工作函数使用单一 `try/except/finally` 生命周期：

- 正常完成、取消、超时和未预期异常都必须且只发送一次完成事件。
- `finally` 必须从 `_tasks` 移除当前任务，避免任务表泄漏。
- 未预期异常写入运行日志；尚未产生终态的歌曲标记为失败，但已经成功、跳过或明确取消的歌曲不改写。
- 保留现有取消语义和单曲成功语义，不因附加元数据失败改成下载失败。

完成事件的组装集中在同一个出口，避免正常路径与异常路径重复发送。

### 2. 浏览器初始化失败清理

增加由 Playwright 工作线程执行的内部清理助手，统一关闭并清空 page、persistent context 等浏览器状态，同时清除 ready 状态。

初始化过程在下列失败点都调用该助手：

- Chrome/persistent context 启动失败；
- 首次页面导航失败；
- Cloudflare 检查或 headed 回退失败；
- 初始化超时或其他未预期异常。

清理必须发生在 Playwright 所属线程，避免跨线程关闭对象。下一次初始化应从无残留状态开始；GUI headless 屏幕外位置参数和 headed 可见回退规则保持不变。

### 3. 保守音频响应校验

校验放在最终文件替换之前、响应体已经可检查之后，按以下顺序处理：

1. 对明确文本错误类型直接拒绝：`text/html`、`application/json`、`application/problem+json`、XML 类型及带相应后缀的类型。
2. 对响应体前缀去除 BOM 和空白后进行大小写不敏感嗅探，拒绝明显 HTML、JSON 或 XML 错误文档。
3. 接受 `audio/*`。
4. 对 `application/octet-stream`、缺失或未知 `Content-Type`，只要内容不像上述错误文档就接受。

校验失败时删除临时文件、记录可理解的失败原因，不覆盖现有最终文件。

不增加严格容器签名或 Mutagen 可解析性校验。部分合法音源可能使用非标准 MIME、重定向下载或容器变体，严格校验的误杀成本高于收益；本方案只阻止已经确认会伪装成音频的常见错误响应。

### 4. 音源目录与展示名全局统一

- `Source` 是唯一手工维护目录，新增 `kugou`。
- `VALID_SOURCES` 从 `Source` 派生，不再单独手写列表。
- 删除 GUI 后端和 Svelte 中的硬编码音源展示映射。
- CLI 参数值、配置值、`Song.source` 和上游请求继续保存原始 API 值。
- 所有面向用户的 CLI/GUI 下拉框、当前设置、搜索提示、结果表格和日志通过统一解析函数显示名称。
- 国内平台使用中文品牌名；国际平台保留通行品牌名。
- 未知音源采用原值回退，以兼容上游扩展。

### 5. GUI 可见搜索反馈

搜索栏增加紧邻输入区的状态反馈区域，用于：

- 空关键词：在前端立即显示简短校验提示，不发起后端搜索。
- 搜索失败：显示面向用户的简短错误信息，详细异常仍进入运行日志。
- 新搜索开始或输入被修正后，清除过期错误。

反馈区域使用现有视觉语言，不改变搜索优先布局，不把底层的 Playwright、Chrome、Cloudflare 或堆栈信息直接展示给用户。

### 6. 搜索完成与结果数量播报

结果区增加适度的 `aria-live="polite"` 状态：

- 搜索开始时播报正在搜索；
- 搜索完成时播报结果数量；
- 无结果和失败使用明确但不重复的文案。

视觉上的总数和已选数仍只在结果区集中显示；不在搜索按钮或下载按钮重复动态数量。播报内容与可见状态共享同一状态来源，避免屏幕阅读器收到互相冲突的消息。

## 数据与兼容性

- CLI 中 `--source netease` 等既有用法完全不变，新增 `--source kugou`。
- GUI 与 CLI 默认音源保持不变。
- 序列化模型继续输出原始 source 值，下载、封面和歌词请求不受中文展示名影响。
- 未知 source 不导致模型解析或界面渲染失败。
- 现有下载文件命名、重复文件判断、取消行为和 warning-only 元数据策略保持不变。

## 测试策略

按测试驱动方式实施，每个问题先增加会失败的回归测试，再写最小修复：

- 枚举/配置测试：13 个音源完整、值唯一、标签非空、`VALID_SOURCES` 精确派生、未知值回退。
- CLI 测试：原始参数值兼容，帮助和面向用户的输出显示统一名称，酷狗可选。
- GUI API/桥接测试：音源选项来自统一目录，日志显示名称，无后端硬编码副本。
- 下载器测试：200 HTML、JSON、XML 被拒绝；audio、octet-stream、缺失/未知 MIME 的非错误内容保守接受；临时文件正确清理。
- 浏览器测试：模拟多个初始化失败点，断言 context/page/ready 状态清空且可重试。
- 下载任务测试：正常、取消、超时和异常均恰好一次完成事件并清理 `_tasks`。
- 前端测试：空关键词和错误可见；结果计数具有礼貌播报；结果行从后端选项解析展示名。
- 构建测试：Svelte 检查、Node 测试及静态产物一致性。

最终验证包含项目规定的 Python 测试、前端测试、Ruff、格式检查、mypy、py_compile、环境检查和前端构建。真实搜索只作为本地集成验证，不加入单元测试；至少验证一个既有音源，并在上游可用时验证酷狗普通搜索。若上游临时不可用，应如实记录，不能把网络波动解释为代码失败或伪造通过结果。

## 文档更新

实现时同步更新：

- `README.md`：支持音源与中文展示说明、酷狗及专辑搜索限制、CLI 示例或参数说明。
- `AGENTS.md`：新增音源只维护 `Source`，其他列表由它派生；保留 GUI 与下载语义约束。
- 必要的既有设计文档或测试说明：只更新与本次行为直接相关的内容。

## 风险与控制

- **上游音源变化**：通过原始值回退避免未知音源破坏 UI；人工目录仍需在确认上游支持后更新。
- **酷狗能力不完整**：明确仅加入可搜索音源，不承诺专辑搜索；不添加未经证实的特殊处理。
- **响应嗅探误判**：只识别明确错误文档，不要求固定音频魔数或 MIME。
- **跨线程浏览器清理**：所有 Playwright 对象在其工作线程创建和销毁。
- **重复完成事件**：完成事件集中到单一出口，并用测试覆盖所有终止路径。
- **前后端目录再次分叉**：前端只消费 GUI API 返回值，测试断言选项由 `Source` 派生。

## 验收标准

1. 项目只有 `Source` 一处需要人工维护音源 ID 和展示名。
2. 13 个确认音源均可通过 CLI/GUI 选择，国内平台均显示中文名，API 收到的仍是原始 ID。
3. 所有面向用户的音源展示路径使用统一名称，未知值安全回退。
4. GUI 下载任务无论如何结束都恰好发送一次完成事件并从任务表移除。
5. 浏览器初始化失败不遗留 persistent context，重试不再因本次失败锁住 profile。
6. 明显 HTML/JSON/XML 假音频不会成为最终文件，宽松音频响应仍保持兼容。
7. 空搜索和搜索失败有可见反馈，搜索完成和结果数量可被辅助技术感知。
8. 既有测试与新增回归测试全部通过，前端静态产物由源码重新构建。
